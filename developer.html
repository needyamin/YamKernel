YamKernel Developer Reference

Current reference for the YamOS v0.4 development tree. YamKernel is a graph-based x86_64 hybrid kernel with Limine boot, YamGraph capabilities, a small userland, and a Wayland-style desktop.

x86_64 Limine YamGraph SYSCALL/SYSRET APIC IPI HPET/TSC VFS XHCI TCP/IP Wayland-style compositor

Architecture

+---------------------------- Ring 3 --------------------------------+
| ELF apps, services, userspace drivers, libc/libgui/libyam           |
+----------------------- SYSCALL / SYSRET ---------------------------+
| file/process/memory/scheduler/Wayland/driver/AI/touch/IPC syscalls  |
+---------------------------- Ring 0 --------------------------------+
| scheduler | VFS | net | USB | input | DRM/compositor | cgroups/OOM  |
+-------------------------- YamGraph --------------------------------+
| tasks, memory, devices, files, namespaces, channels, capabilities   |
+----------------------- Memory / CPU / Boot ------------------------+
| PMM | VMM | guards | ACPI | APIC/IPI | HPET/TSC | SMP | Limine      |
+--------------------------------------------------------------------+

YamGraph is the organizing model: resources are nodes, authority travels as capability-carrying edges, and subsystems can reason about ownership, mappings, dependencies, and channels.

Boot Pipeline

  1. Limine loads the kernel, framebuffer, boot modules, memory map, HHDM, RSDP, and SMP response.
  2. YamBoot presents Normal, Safe Mode, and Reboot choices.
  3. The kernel initializes framebuffer, CPU tables, CPU security features, memory managers, ACPI/APIC, HPET/TSC detection, per-CPU data, SMP, and syscalls.
  4. YamGraph is initialized and core kernel subsystems are registered as graph nodes.
  5. Self-tests run for PMM and YamGraph.
  6. Normal boot starts PCI, USB, VFS, IPC, NET, evdev, and mouse paths. Safe Mode skips this batch.
  7. Scheduler, cgroups, OOM, power, and AI initialize.
  8. PID 1 and the Wayland compositor are spawned. The compositor starts in first-boot setup unless bypassed. The BSP becomes idle.

Syscall ABI

Convention: rax is the syscall number. Arguments are in rdi, rsi, rdx, r10, r8, and r9. Return value is in rax.

The syscall numbers live in src/kernel/api/syscall.h. Userspace wrappers should be kept synchronized with that file.

Memory

Kernel Memory

  • Higher-half kernel and Limine HHDM mapping.
  • PMM initializes from Limine memmap.
  • VMM manages kernel page tables and MMIO mapping windows.
  • Heap and slab provide early and object-style allocation.
  • Scheduler-created kernel stacks use unmapped guard pages on both sides.
  • Page-table changes route through local invalidation plus APIC TLB shootdown hooks.

User Memory

  • ELF app modules are loaded into Ring 3 address spaces.
  • CoW/fork, brk, mmap, munmap, and mprotect hooks are present.
  • User ELF stacks reserve adjacent unmapped guard pages.
  • Wayland buffers map app-visible surfaces to compositor-owned objects.

Scheduler and SMP

The scheduler is CFS-style and includes task sleep, wait queues, mutexes, futexes, process lifecycle hooks, cgroups, OOM scoring, and CPU affinity ABI pieces.

SMP is intentionally conservative right now: Limine brings up APs, each AP initializes GDT/IDT/per-CPU/security/syscall/LAPIC state, then parks. Local APIC IPI delivery and a TLB shootdown vector are available, but full task scheduling on APs is deferred until address-space switching and per-CPU run queue behavior are audited.

Timers and Exceptions

Timers

  • PIT remains available for legacy setup and APIC timer calibration.
  • HPET is parsed from ACPI and initialized when the platform advertises it.
  • TSC, invariant TSC, and TSC-deadline capability are reported during CPU setup.
  • The active scheduler tick still uses the periodic Local APIC timer.

Exceptions

  • IDT exception handlers print CR2, RIP/RSP, flags, and general-purpose registers.
  • The panic layer exposes kpanic_with_frame() for fatal exception register dumps.
  • Page faults first try demand paging and CoW handling before panicking.

Networking

The networking tree contains Ethernet/e1000 integration plus ARP, IPv4, ICMP, UDP, DHCP, DNS, and a TCP state machine. Treat it as active development until QEMU network smoke tests and socket ABI cleanup are complete.

qemu-system-x86_64 -cdrom build/yamkernel.iso -m 256M -smp 2 \
  -netdev user,id=n0 -device e1000,netdev=n0 -serial stdio

Desktop and Userland

Compositor

CPU-rendered Wayland-style compositor with first-boot setup, login state, desktop state, top menu, dock/taskbar, window chrome, app launch, and debug overlay.

Apps

Terminal, calculator, and browser currently launch as compositor-native apps for reliable drawing. Python commands in the main Terminal are reserved for real python.org CPython while the CPython port is wired up. ELF modules for terminal, calculator, browser, Python status, authd, and services are still built into the ISO.

VTTY

Six virtual console buffers with ANSI basics and framebuffer rendering. Alt+F1..F6 is the target switching model.

First-Boot, Login, and Terminal Python

  • Setup asks for computer name, username, and password.
  • Ctrl+Shift+Y bypasses setup/login, creates root/password and guest/guest, logs in root, and opens the desktop.
  • The desktop menu's Python item opens Terminal; normal Python use happens in one Terminal window.
  • Official CPython 3.14.4 source from python.org is vendored at vendor/cpython/Python-3.14.4; YamOS port notes live in src/os/ports/python/cpython.
  • In Terminal, python, python3, py, pip, and pip3 are CPython-only commands. Until CPython is linked, they show CPython port status instead of running a fake interpreter.
  • python -c "print(1+2)", python3 -c ..., py -c ..., and pip install ... are blocked until real CPython can execute inside YamOS.
  • Keyboard modifiers are routed through evdev to focused apps, so shifted characters such as Shift+9 = ( work in Terminal.

Roadmap

v0.4.x

  • Stabilize boot, VFS, syscall ABI, app launch, serial smoke tests.
  • Add explicit IPI/TLB shootdown validation and timer fallback tests.
  • Prove FAT32 and network paths in repeatable QEMU profiles.
  • Make VTTY switching deterministic.

v0.5-v0.7

  • Complete userland process model and shell.
  • Harden storage, USB, networking, and desktop workflows.
  • Expose capability enforcement across drivers and devices.

v1.0

  • Reproducible build and CI smoke tests.
  • Developer docs for syscalls, apps, drivers, and debugging.
  • Clear release ISO with known limitations.
