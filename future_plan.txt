1.1 COMPETITIVE LANDSCAPE (2026)
─────────────────────────────────
  OS              Target Market         Weakness YamOS Can Exploit
  ─────────────── ──────────────────── ────────────────────────────────────
  Linux           Server / Desktop      Complex, bloated for embedded/IoT
  Windows         Desktop / Enterprise  Closed source, heavy resource usage
  Fuchsia (Google)IoT / Mobile          No ecosystem yet, Google-controlled
  Zephyr RTOS     IoT / Embedded        No GUI, no AI, bare minimum
  seL4            Safety-critical       Niche, no desktop/developer story
  Redox           Hobbyist / Research   Slow progress, tiny community
  HarmonyOS       Huawei devices        Vendor-locked, not open
  ReactOS         Windows compat        Legacy focus, no innovation

1.2 MARKET GAPS YamOS CAN FILL
───────────────────────────────
  Gap 1: Lightweight AI-Native OS
    - No existing lightweight OS has kernel-level AI acceleration
    - Edge AI devices need an OS that isn't Linux-heavy
    - YamOS already has tensor memory + compute dispatch in-kernel

  Gap 2: Touchscreen-First Lightweight OS
    - Embedded touchscreen devices (kiosks, POS, medical, automotive)
    - Linux is overkill, Zephyr has no GUI, Android is too heavy
    - YamOS has MT Protocol B + gesture engine + compositor

  Gap 3: Capability-Based Security for IoT
    - IoT devices need fine-grained permissions (not UID/GID)
    - YamGraph capability model is cleaner than Linux LSM
    - Suitable for medical devices, industrial control, automotive

  Gap 4: Developer-Friendly Hobby OS
    - Growing community of OS developers (OSDev, r/osdev)
    - Clean codebase, single Makefile, modern C
    - Good learning platform for kernel internals

1.3 TARGET MARKETS (Priority Order)
─────────────────────────────────────
  [1] Edge AI / IoT Devices      — Smart cameras, robots, drones, sensors
  [2] Embedded Touchscreen        — Kiosks, POS terminals, HMI panels
  [3] Automotive Infotainment     — Dashboard displays, ADAS companions
  [4] Education / Research        — OS courses, kernel research platform
  [5] Lightweight Desktop         — Raspberry Pi, thin clients, old hardware

2.1 CRITICAL (Must Have for v0.4)
──────────────────────────────────
  [C1] SMP — AP cores boot and run tasks (INIT-SIPI-SIPI)
  [C2] Real Filesystem — FAT32 read/write + ext2 read
  [C3] USB Stack — XHCI controller, HID class (keyboard, mouse, touch)
  [C4] Networking — Working TCP/IP stack (not just stubs)
  [C5] POSIX libc — Minimal C library for user apps (printf, malloc, file I/O)
  [C6] Process exec() — Load and run ELF binaries from filesystem
  [C7] Virtual Console — Multiple terminal sessions (Alt+F1..F6)

2.2 IMPORTANT (Should Have for v0.5)
─────────────────────────────────────
  [I1]  GPU Acceleration — Virtio-GPU, basic 2D/3D rendering
  [I2]  Audio Stack — AC97/HDA driver, mixer, PCM playback
  [I3]  Wayland Protocol — Real Wayland protocol (not custom compositor)
  [I4]  Package Manager — Download and install apps from network
  [I5]  Shared Libraries — Dynamic linking (.so support)
  [I6]  User Accounts — Login, password hashing, home directories
  [I7]  Journaling FS — ext4 or custom journaling filesystem
  [I8]  Swap / Paging — Disk-backed virtual memory
  [I9]  ACPI Power — Sleep (S3), Hibernate (S4), proper shutdown
  [I10] Bluetooth Stack — L2CAP, RFCOMM, BLE GATT

2.3 NICE TO HAVE (v0.6+)
─────────────────────────
  [N1]  Container Runtime — Lightweight cgroup+namespace containers
  [N2]  Rust Support — Rust kernel modules or user-space SDK
  [N3]  Vulkan Driver — GPU compute for AI acceleration
  [N4]  ARM64 Port — Raspberry Pi, embedded boards
  [N5]  Secure Boot — UEFI Secure Boot + measured boot chain
  [N6]  eBPF Runtime — In-kernel programmable tracing/networking
  [N7]  WASM Runtime — Run WebAssembly apps natively
  [N8]  NVMe Driver — Fast SSD storage access
  [N9]  Wi-Fi Stack — 802.11 with WPA2/WPA3
  [N10] Camera Driver — V4L2-compatible camera interface for AI vision

═══════════════════════════════════════════
  PHASE 1: v0.4.0 — "Foundation" (8 weeks)
═══════════════════════════════════════════

  Goal: Make YamOS a self-hosting OS that can boot, run programs, and talk to
        the network. This is the minimum viable product for developers.

  Week 1-2: SMP Boot
  ┌─────────────────────────────────────────────────────────────────────┐
  │ • Implement INIT-SIPI-SIPI sequence to wake AP cores               │
  │ • Per-CPU idle tasks on each core                                   │
  │ • Load balancer: migrate tasks across CPU run queues                │
  │ • Per-CPU slab magazines wired to actual CPU IDs                    │
  │ • IPI (Inter-Processor Interrupt) for cross-CPU wakeup              │
  │ Files: cpu/smp.c, sched/sched.c, cpu/apic.c                        │
  └─────────────────────────────────────────────────────────────────────┘

  Week 3-4: Filesystem + exec()
  ┌─────────────────────────────────────────────────────────────────────┐
  │ • FAT32 driver (read/write) — parse BPB, FAT table, directory      │
  │ • VFS layer: mount, open, read, write, close, stat, readdir        │
  │ • Ramdisk (initrd) — embed files into ISO for early boot           │
  │ • exec() syscall — load ELF from VFS, create address space, run    │
  │ • /bin, /etc, /tmp directory structure                              │
  │ Files: fs/fat32.c, fs/vfs.c, fs/initrd.c, sched/user.c            │
  └─────────────────────────────────────────────────────────────────────┘

  Week 5-6: Networking
  ┌─────────────────────────────────────────────────────────────────────┐
  │ • e1000 driver: TX/RX ring buffers, interrupt-driven               │
  │ • Ethernet frame parsing (src/dst MAC, ethertype)                   │
  │ • ARP: request/reply, ARP cache                                     │
  │ • IPv4: header parse/build, basic routing                           │
  │ • ICMP: echo request/reply (ping)                                   │
  │ • UDP: socket create/bind/sendto/recvfrom                           │
  │ • DHCP client: auto-configure IP/gateway/DNS                        │
  │ • TCP: 3-way handshake, data transfer, FIN (simplified)             │
  │ • DNS resolver: query A records                                     │
  │ Files: drivers/net/e1000.c, net/tcp.c, net/udp.c, net/dhcp.c       │
  └─────────────────────────────────────────────────────────────────────┘

  Week 7-8: POSIX libc + USB
  ┌─────────────────────────────────────────────────────────────────────┐
  │ • Minimal libc: stdio (printf, scanf, fopen), stdlib (malloc, free) │
  │ • string.h, ctype.h, errno.h, unistd.h stubs                       │
  │ • USB XHCI controller init + device enumeration                     │
  │ • USB HID class driver (keyboard, mouse, touchscreen)               │
  │ • USB mass storage (read-only, for USB boot media)                  │
  │ Files: userlib/libc.c, drivers/usb/xhci.c, drivers/usb/hid.c       │
  └─────────────────────────────────────────────────────────────────────┘

  PART 4: DIFFERENTIATION STRATEGY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  What Makes YamOS Different From Every Other OS:

  1. YAMGRAPH RESOURCE MODEL
     - Not UID/GID (Linux), not ACLs (Windows), not static caps (seL4)
     - Live directed graph where permissions flow as edges
     - Revocation cascades through graph topology
     - Scheduler and IPC reason about resource relationships

  2. KERNEL-NATIVE AI
     - Only OS with tensor memory management in the kernel
     - DMA-pinned, cache-aligned tensor allocations
     - Zero-copy inference pipeline: sensor → kernel → result
     - No userspace framework overhead (no PyTorch/TensorFlow needed)

  3. TOUCHSCREEN-FIRST DESIGN
     - MT Protocol B in the kernel (not bolted on later)
     - Gesture recognition engine with configurable thresholds
     - Palm rejection, calibration, pressure sensitivity
     - Direct compositor integration (no X11/Wayland middleware)

  4. LIGHTWEIGHT + MODERN
     - Boots in <2 seconds on modern hardware
     - <16MB RAM for kernel + basic apps
     - Modern C (no legacy POSIX baggage)
     - Clean, readable codebase (~15K lines for full kernel)
