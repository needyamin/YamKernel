YamOS Future Plan
Version: v0.4.0 development tree -> v1.0.0
Updated: May 2026

===============================================================================
CURRENT BASELINE
===============================================================================

YamOS has moved past the original v0.3 foundation. The tree now includes:

- Limine BIOS/UEFI boot, YamBoot, splash assets, and ELF module loading.
- Core x86_64 kernel setup: GDT, IDT, TSS, CPUID, security flags, APIC/IOAPIC.
- Limine SMP startup with AP initialization and parking.
- PMM, VMM, heap, slab, CoW/fork support, brk/mmap/mprotect hooks.
- CFS-style scheduler, wait queues, mutex/futex, cgroups, OOM, power hooks.
- YamGraph resources, channels, and capabilities.
- VFS with initrd root, devfs, procfs, FAT32 driver code, ELF loader.
- USB XHCI, USB core, HID, input, evdev, touch, gesture, mouse, keyboard paths.
- e1000 and in-tree ARP/IP/ICMP/UDP/TCP/DHCP/DNS networking code.
- Wayland-style compositor with login, top menu, dock/taskbar, app launch, VTTY mode.
- Userland app modules and minimal libc/libgui/libyam support.

The main remaining theme is not "add everything from scratch"; it is to make the
existing in-tree systems reliable, integrated, testable, and useful on real
hardware and in repeatable QEMU scenarios.

===============================================================================
PHASE 1: v0.4.x STABILIZATION
===============================================================================

Goal: turn the v0.4 feature set into a dependable developer preview.

1. Boot and scheduler correctness
   - Keep Limine SMP as the primary path.
   - Audit AP address-space switching before enabling task scheduling on APs.
   - Add scheduler assertions around current task, run queue ownership, and fd tables.
   - Add a boot-mode flag for "single core debug" versus "AP initialized and parked".

2. Syscall ABI cleanup
   - Synchronize `src/kernel/api/syscall.h` with `src/os/lib/libyam/syscall.h`.
   - Fix stale wrappers that reference names not dispatched by the kernel.
   - Add syscall return conventions and errno mapping to developer docs.
   - Add a small ABI smoke app that exercises open/read/write/close/getpid/sleep.

3. VFS and initrd reliability
   - Add path normalization and mount-boundary checks.
   - Wire `stat`, `readdir`, and `lseek` consistently.
   - Make `/dev/null`, `/dev/tty*`, `/proc/cpuinfo`, and `/proc/meminfo` stable.
   - Add file descriptor inheritance rules for fork/exec.

4. FAT32 integration
   - Mount a real FAT32 disk image in QEMU.
   - Add read/write/readdir tests against known image contents.
   - Add short-name and long-name behavior notes.
   - Add bounds checks around cluster traversal and disk image size.

5. Networking smoke path
   - Create a QEMU e1000 target with `-netdev user`.
   - Prove DHCP -> gateway -> ICMP ping -> UDP DNS -> TCP connect.
   - Add packet counters and ring-state diagnostics to the shell/procfs.
   - Harden TCP timeout and reset paths.

6. Desktop and VTTY polish
   - Make Alt+F1..F6 switching deterministic.
   - Keep compositor state and VTTY state explicit: login, desktop, VTTY.
   - Ensure app module launch failures show a visible notification/log.
   - Add a lower-resolution fallback for slower VMs.

7. Documentation and scripts
   - Keep README, DEBUGGING, developer.html, and this roadmap aligned.
   - Add repeatable smoke commands for BIOS, UEFI, serial-only, network, and USB.
   - Add a short "known limitations" section to release notes.

Exit criteria:

- `make iso` succeeds on a clean Ubuntu/WSL environment.
- BIOS and UEFI QEMU boot reach PID 1 and compositor or Safe Mode shell.
- Serial log includes successful VFS, scheduler, and compositor initialization.
- At least one user ELF app launches from the desktop.
- A headless smoke run can be repeated without manual intervention.

===============================================================================
PHASE 2: v0.5 DESKTOP AND USERLAND
===============================================================================

Goal: make YamOS feel like a small usable OS, not just a kernel demo.

1. Userland process model
   - VFS-backed `exec(path, argv, envp)`.
   - `waitpid`, process exit status, parent/child lifecycle, PID 1 reaping.
   - Basic signal delivery rules.
   - Process list in `/proc`.

2. libc and shell
   - Complete stdio basics: fopen/fread/fwrite/fclose/fgets/fprintf.
   - Align unistd wrappers with kernel syscall numbers.
   - Add a small shell with `ls`, `cat`, `echo`, `ps`, `mount`, `uname`, `reboot`.
   - Provide stable app headers for GUI and IPC apps.

3. Desktop applications
   - Terminal using VTTY or Wayland surface backend.
   - Calculator, browser demo, Python demo, image viewer.
   - Auth/login flow with a minimal `/etc/passwd` model.
   - Notifications for app launch, crash, and network status.

4. Graphics
   - Reduce full-screen redraw cost.
   - Add dirty rectangle composition.
   - Add cursor damage tracking.
   - Investigate virtio-gpu or a cleaner DRM abstraction.

Exit criteria:

- User can log in, open a terminal, run shell commands, launch apps, and switch VTTYs.
- User apps communicate through stable syscalls and libraries.
- Desktop survives app crashes without taking down the compositor.

===============================================================================
PHASE 3: v0.6 HARDWARE AND NETWORKING
===============================================================================

Goal: broaden hardware coverage and make networked workflows real.

1. Storage
   - AHCI or NVMe driver.
   - Block device layer.
   - Mountable FAT32 disk image and read-only ext2 target.
   - Optional write-ahead journal for YamOS-native experiments.

2. Network
   - Harden e1000 TX/RX descriptors and interrupts.
   - Add socket syscalls for TCP/UDP.
   - HTTP client demo.
   - DNS cache and DHCP renewal.
   - Better error codes and nonblocking mode.

3. USB
   - Complete enumeration state machine.
   - Mass storage class.
   - HID boot protocol fallback for keyboards and mice.
   - Hotplug diagnostics.

4. Audio and Bluetooth
   - AC97/HDA probe and PCM playback.
   - Mixer device.
   - Bluetooth HCI exploration, then L2CAP/RFCOMM later.

Exit criteria:

- YamOS can boot with virtual disk + virtual NIC + USB keyboard/mouse in QEMU.
- Network socket demo connects to a host service.
- USB keyboard/mouse works through the USB path, not only PS/2.

===============================================================================
PHASE 4: v0.7 SECURITY AND ISOLATION
===============================================================================

Goal: make the YamGraph capability model visible and enforceable.

1. Capabilities
   - Capability checks on file, device, channel, MMIO, and driver syscalls.
   - Revocation behavior documented and tested.
   - Capability audit logs.

2. Driver isolation
   - Move selected drivers toward user-mode service model.
   - Gate I/O port, MMIO, IRQ, and PCI config access through capabilities.
   - Crash containment for user-mode drivers.

3. Hardening
   - Stack canaries.
   - W^X policy.
   - KASLR investigation.
   - Signed kernel/module verification plan.

Exit criteria:

- A user process cannot open devices or map MMIO without a capability.
- Revoking a capability blocks future operations and appears in audit logs.
- Driver crash does not corrupt the kernel or compositor.

===============================================================================
PHASE 5: v0.8 AI, TOUCH, AND EDGE DEVICE STORY
===============================================================================

Goal: sharpen the project differentiators.

1. AI acceleration
   - Stable tensor object lifecycle.
   - Device query and job dispatch demos.
   - Optional ONNX subset parser or a tiny model format.
   - Zero-copy sensor -> tensor -> result experiment.

2. Touch
   - MT Protocol B validation.
   - Calibration persistence.
   - Gesture configuration UI.
   - Kiosk-mode shell.

3. Edge profile
   - Fast boot profile.
   - Read-only root with writable data partition.
   - Recovery boot mode.
   - Watchdog support.

Exit criteria:

- Demo shows touchscreen-first UI and an AI job path.
- Edge/kiosk profile boots into a constrained app with visible health status.

===============================================================================
PHASE 6: v1.0 RELEASE
===============================================================================

Goal: a clear, reproducible release for developers and OS research.

1. Build and CI
   - Reproducible toolchain container or script.
   - CI build for kernel and ISO.
   - Headless QEMU smoke tests.
   - Artifact publishing.

2. Test coverage
   - Kernel unit tests for pure helpers.
   - Boot smoke tests for BIOS/UEFI.
   - Filesystem image tests.
   - Network packet tests where feasible.

3. Documentation
   - Developer architecture guide.
   - Syscall ABI reference.
   - App development guide.
   - Driver development guide.
   - Debug cookbook.

4. Release packaging
   - v1.0 ISO.
   - Demo video/script.
   - Known hardware and QEMU profiles.
   - Security and limitations statement.

Exit criteria:

- A new contributor can build, boot, debug, and run a demo app from the docs.
- The release has known-good QEMU commands and reproducible artifacts.
- Limitations are explicit instead of hidden.

===============================================================================
MARKET AND POSITIONING
===============================================================================

Best-fit audiences:

1. OS development learners who want a modern C/ASM x86_64 kernel with GUI/userland.
2. Kernel research experiments around graph-based capabilities.
3. Edge AI and touchscreen/kiosk prototypes.
4. Hobby desktop OS explorers.

Differentiators:

- YamGraph resource model instead of only UID/GID-style ownership.
- Kernel-visible AI/tensor abstractions.
- Touch and GUI paths are first-class project goals.
- Small, readable, single-repo OS stack.

Risks:

- Hardware coverage is tiny compared with Linux/BSD.
- Networking, USB, FAT32, and desktop code need much more testing.
- Multi-core scheduling is not enabled yet despite AP initialization.
- Userland ABI must be cleaned before external app development is pleasant.

Near-term priority:

Stabilize the v0.4 tree before adding large new subsystems. The fastest path to a
credible YamOS is repeatable boot, working docs, a reliable shell/app loop, and a
small set of demos that do exactly what they claim.
