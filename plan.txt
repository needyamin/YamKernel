YamOS Development Roadmap & Market Analysis
===========================================

1. Market Positioning
---------------------
YamOS occupies a unique niche in the hobbyist OS landscape, prioritizing high-fidelity 
visual experiences and macOS-inspired workflows.

Comparison:
- SerenityOS: Unix from scratch | Strength: Custom Ecosystem | Gap: Visual Modernity
- Haiku: BeOS Successor | Strength: Usability | Gap: Modern Web Support
- Redox: Rust Safety | Strength: Security | Gap: Application UX
- YamOS: Apple-like UI | Strength: Aesthetics | Gap: Kernel Infrastructure

2. Critical Technical Gaps
--------------------------
- Threads & Signals: MISSING. Essential for modern language runtimes and 
  multi-process browsers. (Target: pthread-compatible ABI).
- Dynamic Linking: MISSING. ELF interpreter required for shared libraries 
  to reduce memory pressure.
- Asynchronous I/O: MISSING. Necessary for responsive network applications 
  (Target: select/poll/epoll).
- Page Cache: [DONE] 1MB LRU Block Cache integrated into VFS with dirty-tracking.
- Persistent Storage: [DONE] FAT32 Write-back with AHCI/VirtIO-blk support.

3. Implementation Roadmap
-------------------------

Phase 1: The Runtime Foundation [IN PROGRESS]
- Kernel Threads & Futexes (Internal core present, needs public ABI)
- Build ELF Dynamic Loader (ld.so)
- Standard POSIX Signal Delivery

Phase 2: Performance & Hardware [DONE]
- 1MB LRU Block Cache for VFS
- AHCI SATA Driver (DMA-enabled)
- DRM Modesetting & Damage Tracking (Bochs VBE)
- Runtime Resolution Switching (1024x768 -> 1920x1080)

Phase 3: Connected Ecosystem [NEXT]
- Non-blocking Socket ABI (select/poll)
- Full TLS/HTTPS Stack (Porting mbedTLS or similar)
- Initial Package Manager (ypkg) & Downloader

Phase 4: Experience Polish
- Hardware-Accelerated Blurs (using Dirty Rects)
- FreeType / HarfBuzz Font Integration
- VFS-level Permissions (UID/GID enforcement)
