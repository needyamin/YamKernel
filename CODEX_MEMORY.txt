==================

Read this first before working on the repo.

Project

- Main ISO: build/yamkernel.iso

Owner Direction
---------------
- Keep the repo focused on kernel and OS-level work only.
- Standing implementation rule: when any requested feature depends on missing
  or incomplete kernel support, update the kernel-level foundation first
  (driver, memory, VFS, syscall ABI, scheduler, security, network, storage, or
  process model as appropriate), verify it, and only then build the app/service
  feature on top. Do not paper over missing kernel infrastructure with
  application-level theater.
- Focus on serious kernel development, driver development, memory management,
  networking, filesystems, process model, and hardware support.
- Do not add fake language runtimes, package-manager theater, or language-
  specific installer behavior.
- Any installer/capability work should be generic OS infrastructure only.
- YamOS is not complete enough to run arbitrary x86_64 OS software yet. Standing
  required foundations:
  - persistent disk/storage driver fully mounted as writable system volume
  - full POSIX/Linux syscall compatibility
  - `execve` and stronger process model
  - dynamic linker / shared libraries
  - threads and signals
  - real permissions/users/security
  - TLS/HTTPS + certificate store
  - nonblocking sockets, `select`/`epoll`
  - package manager/downloader
  - full browser engine support
  - language runtime porting layer
  - real hardware driver coverage beyond current QEMU-focused path
- Missing compatibility details to track:
  - process execution: `execve`, argv/envp, interpreter scripts, wait/exit
  - dynamic linking: `PT_INTERP`, relocations, `dlopen/dlsym`, library paths
  - threads: pthread ABI, TLS, thread-local errno, robust futex
  - signals: delivery, masks, handlers, default actions, interruption semantics
  - files: persistent root, rename/link/symlink/readlink, stat accuracy,
    fcntl, locks, permissions
  - paths: per-process cwd, chdir/getcwd, openat family
  - memory: file-backed mmap, shared mappings, msync/mremap
  - sockets: nonblocking, select/poll/epoll, sockopts, peer names, UDP, IPv6
  - time: monotonic/realtime clocks, nanosleep, timers, timezone/localtime
  - security: UID/GID, file modes, capabilities, sandboxing, secure random
  - terminal: PTY, termios, line discipline, job control
  - packages: HTTPS downloader, certificates, signatures, install database
  - browser engine: TLS, DOM, CSS layout, JS hosting, codecs, cache/cookies,
    storage, sandboxing
  - hardware: storage, GPU, USB classes, audio, Wi-Fi, power management

  .\scripts\test.ps1 headless
  .\scripts\test.ps1 verify
  .\scripts\test.ps1 debug
- Owner preference for verification:
  - Do not run raw qemu-system-x86_64 directly every time.
  - Prefer project debug/log helpers such as .\scripts\test.ps1 verify,
    .\scripts\test.ps1 debug, .\scripts\test.ps1 serial, or existing debug logs.
  - Use direct QEMU only when a specific hardware configuration is required and
    the helper cannot express it, then explain why.

    policy. Use:
    powershell -ExecutionPolicy Bypass -File .\scripts\test.ps1 verify
