Recent Python Terminal Work
---------------------------
- IMPORTANT CURRENT TRUTH:
  - YamOS does NOT have real CPython like Windows/Linux yet.
  - User explicitly wants python.org real Python, i.e. CPython, not MicroPython
    and not YamPy/fallback.
  - Official CPython 3.14.4 source has been downloaded into
    vendor/cpython/Python-3.14.4.
  - YamOS-specific CPython notes exist at src/os/ports/python/cpython.
  - Terminal `python`, `python3`, `py`, `pip`, and `pip3` are now CPython-only
    command names. Until CPython is linked, they show CPython port status and
    do not execute fake Python/pip code.
  - User wants auto-download/install if missing and full pip/data science later.
    Current YamOS lacks HTTPS/TLS downloader, persistent writable system volume,
    complete POSIX/libc, and linked CPython, so this must be implemented as OS
    runtime work, not terminal fakery.
  - Real Python target is now CPython from python.org. It still needs a larger
    YamOS POSIX/libc/filesystem/tty/process/time compatibility layer before it
    can execute.
- src/os/apps/terminal.c:
  - Older userspace ELF terminal no longer exposes a Python-like REPL under
    python/python3/py. Those names are CPython-only there too.
- src/os/apps/python.c:
  - Python app is now a CPython port-status UI, not a fake downloader.
- Limitation:
  - This is not full CPython yet. CPython source is present, but it is not
    linked into build/python.elf yet.
- Build note:
  - `make iso` passed after removing fallback parser exposure and MicroPython
    source/scaffold.
  - `timeout 55s make run` passed after CPython-only switch. Boot reached

    getenv/setenv/unsetenv, integer conversions, getcwd/chdir/access/isatty,

  - `make iso` passed after adding posix.c/time.c to LIBC_SRCS.
  - `timeout 55s make run` passed after libc/POSIX slice. Boot reached desktop,
    Terminal opened, `python` path still reports CPython status, and no panic

  - Latest terminal fix: `python`, `python3`, `py`, `pip`, and `pip3` in the
    compositor-native terminal now call `cmd_python_install()` and launch
    `/boot/python.elf` through `wl_spawn_app_async(...)` instead of only

  - `make iso` passed after the terminal installer-launch change.
  - First `timeout 55s make run` after that showed terminal successfully
    launching `python-installer`, but the installer crashed with GPF #13 at
    `0x4011f4`. Cause: ELF userspace entered with 16-byte aligned RSP, while
    GCC-generated x86_64 C function entry expects RSP % 16 == 8; compiler
    emitted `movaps` to an unaligned stack slot.
  - Fixed in `src/fs/elf.c`: enter userspace with `USER_STACK_TOP - 8` to match
    the SysV x86_64 stack ABI for current C `_start` apps. Also moved the
    Python installer step string table in `src/os/apps/python.c` out of the
    local stack.
  - `make iso` passed after the ELF stack-alignment fix.
  - `timeout 55s make run` passed after the fix: logs showed
    `[PYTHON] installer launched from terminal`, `python-installer` ELF load,
    `[APP_DBG] CPython installer start`, `map-ok`, `first-content`, repeated
    `[PYTHON] checking CPython installer requirements`, and final
    `[PYTHON] CPython installer blocked: linker/VFS/HTTPS/pip storage required`.
    No kernel panic occurred; QEMU exited nonzero only because timeout stopped
    it.
  - Follow-up after user saw "CPython installer blocked by missing OS service":
    implemented real kernel/VFS/libc `dup` and `dup2` support:
    `src/fs/vfs.c`, `src/fs/vfs.h`, `src/cpu/syscall.c`,
    `src/os/lib/libc/unistd.h`, and `src/os/lib/libc/posix.c`.
    This removes one actual CPython POSIX blocker instead of hiding it.
  - `src/os/apps/python.c` now probes `SYS_DUP` and `SYS_DUP2` at runtime and
    logs `[PYTHON] POSIX dup/dup2 service ready` when they work. Its final
    status no longer says the vague "missing OS services"; it now says the next
    blocker is writable Python `Lib/` path plus static CPython interpreter link.
  - `make iso` passed after adding dup/dup2.
  - `timeout 55s make run` passed after adding dup/dup2. Logs showed:
    `[VFS] dup fd1 -> fd3 path='/dev/console' refs=1`,
    `[VFS] dup2 fd1 -> fd9 path='/dev/console' refs=1`,
    `[PYTHON] POSIX dup/dup2 service ready`, and
    `[PYTHON] CPython next blocker: writable Lib path + static interpreter link`.
    No kernel panic occurred; QEMU exited nonzero only because timeout stopped

Recent First-Boot / Multi-User Work
-----------------------------------
- Compositor now starts in COMPOSITOR_STATE_SETUP.
- Setup asks for:
  - Computer name
  - Username
  - Password
- Keyboard:
  - Tab changes focused field.
  - Enter creates account.
  - Backspace edits current field.
- Mouse:
  - Clicking setup fields changes focus.
- Accounts:
  - In-memory account table added in wl_compositor_t.
  - First setup user is admin and becomes current session user.
  - Built-in test accounts are added after setup:
    root / password
    guest / guest
- Login:
  - Login checks username + password against account table.
  - Menubar shows current_user@computer_name.
  - Launcher has "Lock / Switch User" to return to login.
- Blank-screen follow-up:
  - Setup is now rendered as an overlay on top of normal desktop chrome instead
    of replacing the whole desktop render path.
  - Ctrl+Shift+Y skips setup/login, creates default users root/password and
    guest/guest, logs in root, and opens desktop.
  - Setup overlay text shows Ctrl+Shift+Y bypass instruction.
- App blank follow-up:
  - Terminal, Browser, and Calculator launcher/menu entries now start the
    compositor-native kernel tasks (wl_term_task, wl_browser_task,
    wl_calc_task) instead of the ELF userspace apps. This avoids blank app
    windows when the userspace shared-buffer mapping path misbehaves.
  - The ELF apps still build into the ISO; the shared-buffer path needs a
    separate investigation before switching launchers back to them.
- Terminal Python follow-up:
  - Because the visible Terminal is now compositor-native, Python command
    support also exists in src/os/services/compositor/wl_terminal.c.
  - In that terminal, python/python3/py/pip/pip3 are CPython-only names. They
    launch `/boot/python.elf` installer/status app and show status until
    CPython is linked. python -c / python3 -c / py -c and pip install are
    blocked until real CPython/pip can execute.
  - The desktop Python launcher/menu now opens Terminal for the python command
    instead of launching the separate python.elf window.
  - Ctrl+Shift+Y is the setup/login bypass shortcut. It creates default users
    root/password and guest/guest, logs in root, and opens the desktop.
  - Important fix: keyboard.c must push Shift press events to evdev, otherwise
    compositor_input.c never sees shift_held=true and Ctrl+Shift+Y will not
    trigger.
  - Important fix: compositor_input.c must not consume Shift/Ctrl/Alt modifier
    events after updating global modifier state. Let them route to the focused
    app so Terminal can produce shifted characters like Shift+9 = '('.
- Wayland/userspace buffer follow-up:
  - Python ELF window previously blanked because WL_MAP failed with
    "surface N not found"; src/cpu/syscall.c now falls back to the newest active
    surface owned by the same task for WL_MAP/WL_COMMIT if direct ID lookup
    misses.
  - Debug log after fix showed python map-ok and first-content checksum.
- Keyboard/modifier follow-up:
  - Shifted symbols in Terminal should work: Shift+9='(', Shift+0=')',
    Shift+1='!', uppercase letters, etc.
  - keyboard.c now pushes Shift press events to evdev.
  - compositor_input.c tracks Ctrl/Shift/Alt globally but does not swallow
    those events, so focused apps also receive modifier state.
- Limitation:
  - Accounts are RAM-only right now. Since initrd root is read-only, reboot
    returns to first-boot setup. Next step is persistent /etc/passwd-style or
    registry-style account storage on writable FAT/disk.

Next Likely Improvements
------------------------
- Add persistent user/computer config storage:
  - Decide where writable system config lives, e.g. /etc/yamos/users.db or
    /var/lib/yamos/accounts.
  - Need writable VFS mount before compositor init can load/save.
  - Store password hashes instead of plaintext once hash helpers exist.
- Add UI for creating more users after setup.
- Make authd verify username + password or retire authd fallback path.
- Add proper session ownership to apps/tasks once scheduler has user IDs.
- Implement real Python:
  - Continue CPython port from src/os/ports/python/cpython using source in
    vendor/cpython/Python-3.14.4.
  - Keep output as build/python.elf.
  - Need enough blocking tty/stdin, libc/syscall, VFS, process, time, env, and
    POSIX compatibility for useful Python behavior.
  - `dup`/`dup2` are now implemented and tested through the Python installer
    readiness probe.
  - For auto-download/install and pip: need HTTPS/TLS, certificate handling,
    writable persistent package storage, CPython Lib install, ensurepip/pip,
    and eventually package build/native extension support.
  - Userland floating point was enabled in USER_CFLAGS with `-msse2
    -mfpmath=sse` while kernel CFLAGS remain no-SSE. libc now has strtod().
    Scheduler already saves/restores FPU/XSAVE state, but this needs deeper
    floating-point user-app testing before CPython/data science can rely on it.
  - Dedicated Makefile rule now compiles src/os/lib/libc/*.c with USER_CFLAGS,
    not kernel no-FPU CFLAGS.
  - ELF userspace now enters with `USER_STACK_TOP - 8` so C `_start` code gets
    SysV-style RSP alignment. Keep this in mind for CPython and other SSE-using
    apps.
  - `make iso` passed after enabling userland SSE2/SSE math and strtod().
  - `timeout 55s make run` reached desktop and Python terminal path with no
    panic/invalid-op fault; command exited nonzero only because timeout stopped
    QEMU.
  - Dynamic extension modules can wait; first target should be a static CPython
    interpreter with a limited built-in module set.
- Optional: switch Terminal/Browser/Calculator back to ELF userspace apps after
  the shared-buffer path is proven stable in repeatable QEMU tests.
